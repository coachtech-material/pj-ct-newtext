# Tutorial 4-4-3: 変更のアップロード (git push)

## 🎯 このセクションで学ぶこと

*   `git push` コマンドを使って、ローカルリポジトリのコミット履歴を、リモートリポジトリにアップロードする方法を学ぶ。
*   `git push origin main` というコマンドの、各部分が何を意味するのかを理解する。
*   `push` が成功したことを、GitHubのWebサイト上で確認する方法を学ぶ。

---

## 導入

ついに、この時が来ました。あなたは、ローカルリポジトリで、いくつかのコミットを積み重ね、`git remote add` によって、その歴史を共有するための「共有の場所（リモートリポジトリ）」への道も確保しました。

今から行うのは、その道を実際に使って、あなたの手元にあるローカルリポジトリの歴史（コミット）を、GitHub上のリモートリポジトリに、**送り届ける**作業です。この行為を、Gitでは **プッシュ (Push)** と呼びます。そして、そのためのコマンドが `git push` です。

`push` を行うことで、初めて、あなたのコンピュータの中だけで完結していた作業が、インターネットを通じて他の人と共有可能になります。チーム開発の第一歩であり、あなたのコードが世界に公開される（あるいは、チーム内で共有される）瞬間です。

---

## 詳細解説

### 🚀 `git push` で歴史を送り届ける

ローカルのコミットを、リモートにアップロードするための基本的なコマンドは、以下の形式です。

```bash
git push <リモート名> <ブランチ名>
```

*   `git push`: ローカルの変更をリモートに送り届ける、という宣言。
*   `<リモート名>`: どのリモートリポジトリに送るかを指定します。前のセクションで設定したニックネーム、つまり `origin` を指定します。
*   `<ブランチ名>`: どの「歴史の流れ（ブランチ）」を送るかを指定します。現時点では、あなたは `main` という、Gitが自動的に作成した、ただ一つのブランチしか使っていません。したがって、`main` を指定します。（ブランチについては、後のチュートリアルで詳しく学びます）

つまり、今回実行するコマンドは、以下のようになります。

```bash
git push origin main
```

このコマンドは、「**`main` ブランチの歴史を、`origin` という名前のリモートリポジトリに、`push`（送り届け）してください**」という意味になります。

### 実行してみよう

ターミナルで、`my-first-git` ディレクトリの中にいることを確認し、以下のコマンドを実行してください。

```bash
git push origin main
```

初めてGitHubに接続する場合、OSによっては、ユーザー名とパスワード（または、アクセストークン）を尋ねるダイアログが表示されることがあります。その場合は、あなたのGitHubの認証情報を入力してください。

成功すると、Gitは以下のようなメッセージを返します。

```
Enumerating objects: 6, done.
Counting objects: 100% (6/6), done.
Delta compression using up to 8 threads
Compressing objects: 100% (4/4), done.
Writing objects: 100% (6/6), 639 bytes | 639.00 KiB/s, done.
Total 6 (delta 0), reused 0 (delta 0), pack-reused 0
To https://github.com/your-name/my-first-git.git
 * [new branch]      main -> main
```

このメッセージは、ローカルリポジトリにあるオブジェクト（コミットやファイルなど）を、圧縮して、リモートリポジトリに書き込んだことを示しています。最後の `* [new branch] main -> main` という行は、リモートリポジトリに、`main` という新しいブランチが作成され、ローカルの `main` ブランチの内容が、そこにコピーされたことを意味します。

### ✅ GitHubで確認する

本当に `push` が成功したか、GitHubのWebサイトで確認してみましょう。あなたのリモートリポジトリのページ（`https://github.com/your-name/my-first-git`）を、ブラウザで再読み込みしてください。

どうでしょうか？

以前は、「Quick setup」の案内が表示されていただけの空っぽのページが、あなたのローカルリポジトリと全く同じ状態になっているはずです。

*   `README.md` と `LICENSE` ファイルが表示されている。
*   最新のコミットメッセージ（`Update README.md`）が表示されている。
*   コミットの数をクリックすれば、ローカルで `git log` を実行した時と同じ、コミット履歴を閲覧できる。

<!-- ここにスクリーンショットを挿入: push成功後のGitHubリポジトリページ -->

これで、あなたのローカルでの作業履歴が、無事にクラウド上のGitHubにバックアップされ、共有可能な状態になりました。

> **ヒント**: 2回目以降の `push` では、`git push` とだけ入力しても、Gitが賢く判断して、`origin` の `main` ブランチに `push` してくれる場合があります。これは、最初の `push` 時に、Gitが自動的に「上流ブランチ (upstream branch)」という設定を行ってくれるためです。しかし、慣れないうちは、混乱を避けるためにも、`git push origin main` と、送り先を明示的に指定する癖をつけておくと良いでしょう。

## ✨ まとめ

このセクションでは、`git push` コマンドを使って、ローカルリポジトリの変更を、リモートリポジトリにアップロードする方法を学びました。

*   **目的**: ローカルでのコミット履歴を、GitHub上のリモートリポジトリに送り届け、共有・バックアップする。
*   **コマンド**: `git push <リモート名> <ブランチ名>`
*   **具体的なコマンド**: `git push origin main` （`main` ブランチの歴史を、`origin` に `push` する）
*   **確認**: `push` 後、GitHubのWebページを再読み込みして、ローカルリポジトリと同じ状態になっていることを確認する。

`push` は、あなたのローカルでの地道な作業を、チームや世界と繋げるための、非常に重要なステップです。これからは、「キリの良いところまで作業が進んだら `commit` し、一日の作業の終わりには `push` する」というサイクルを、習慣にしていきましょう。

次のセクションでは、`push` とは逆の操作、つまり、リモートリポジトリでの変更を、ローカルリポジトリに取り込む方法を学びます。
