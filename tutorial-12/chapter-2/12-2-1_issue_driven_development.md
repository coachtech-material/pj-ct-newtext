## 導入：イシュー駆動開発とは何か？

**イシュー駆動開発（Issue-Driven Development）**とは、**すべての作業をイシュー（課題）として管理し、イシューごとにブランチを作成して開発を進める手法**です。

個人開発では、「今日は何を作ろうかな」と思いつきで開発を進めることができます。しかし、チーム開発では、**誰が何を作っているのか**を明確にする必要があります。

イシュー駆動開発を使うと、以下のメリットがあります。

*   **作業の可視化**: 誰が何を作っているのかが一目でわかる
*   **作業の分担**: イシューごとに担当者を割り当てられる
*   **進捗管理**: イシューの状態（未着手、進行中、完了）を管理できる
*   **履歴の記録**: イシューに議論やコミットを紐づけられる

---

## 詳細解説

### 🔍 イシュー駆動開発の流れ

1. **イシューを作成**: 新機能やバグ修正をイシューとして登録
2. **ブランチを作成**: イシュー番号を含むブランチを作成（例：`feature/1-add-login`）
3. **開発**: ブランチで開発を進める
4. **プルリクエストを作成**: 開発が完了したら、プルリクエストを作成
5. **コードレビュー**: チームメンバーがコードをレビュー
6. **マージ**: レビューが完了したら、`main`ブランチにマージ
7. **イシューをクローズ**: イシューを完了としてクローズ

---

### 🚀 実践例1: イシュー駆動開発の流れ

#### ステップ1: イシューを作成

GitHubで新しいイシューを作成します。

**タイトル**: ログイン機能の実装

**説明**:

```
## 概要
ユーザーがログインできる機能を実装する

## 詳細
- メールアドレスとパスワードでログインできる
- ログインに成功したら、ダッシュボードにリダイレクトする
- ログインに失敗したら、エラーメッセージを表示する

## タスク
- [ ] ログインフォームの作成
- [ ] ログイン処理の実装
- [ ] バリデーションの実装
- [ ] テストの作成
```

イシュー番号は`#1`とします。

---

#### ステップ2: ブランチを作成

```bash
git switch main
git pull origin main
git switch -c feature/1-add-login
```

ブランチ名に**イシュー番号**を含めます。

---

#### ステップ3: 開発

```bash
# コードを書く
git add .
git commit -m "Add login form #1"
git push origin feature/1-add-login
```

コミットメッセージに**イシュー番号**を含めます。

---

#### ステップ4: プルリクエストを作成

GitHubでプルリクエストを作成します。

**タイトル**: ログイン機能の実装 #1

**説明**:

```
## 概要
ログイン機能を実装しました

## 変更内容
- ログインフォームの作成
- ログイン処理の実装
- バリデーションの実装
- テストの作成

Closes #1
```

`Closes #1`と書くと、プルリクエストがマージされたときに、**イシュー#1が自動的にクローズ**されます。

---

#### ステップ5: コードレビュー

チームメンバーがコードをレビューします。

---

#### ステップ6: マージ

レビューが完了したら、`main`ブランチにマージします。

---

#### ステップ7: イシューをクローズ

プルリクエストがマージされると、イシュー#1が自動的にクローズされます。

---

### 💡 TIP: イシューとプルリクエストの紐づけ

コミットメッセージやプルリクエストの説明に、以下のキーワードを含めると、イシューと紐づけられます。

*   `Closes #1`: イシュー#1をクローズ
*   `Fixes #1`: イシュー#1を修正
*   `Resolves #1`: イシュー#1を解決

---

### 🔍 イシュー駆動開発のメリット

#### 1. 作業の可視化

イシューを見れば、**誰が何を作っているのか**が一目でわかります。

---

#### 2. 作業の分担

イシューごとに担当者を割り当てられるので、**作業の重複を防げます**。

---

#### 3. 進捗管理

イシューの状態（未着手、進行中、完了）を管理できるので、**プロジェクトの進捗を把握できます**。

---

#### 4. 履歴の記録

イシューに議論やコミットを紐づけられるので、**なぜその変更を行ったのか**を後から確認できます。

---

### 🚀 実践例2: バグ修正のイシュー駆動開発

#### ステップ1: イシューを作成

GitHubで新しいイシューを作成します。

**タイトル**: ログインエラーが表示されない

**説明**:

```
## 概要
ログインに失敗したときに、エラーメッセージが表示されない

## 再現手順
1. ログインフォームにアクセス
2. 間違ったメールアドレスとパスワードを入力
3. ログインボタンをクリック

## 期待する動作
エラーメッセージが表示される

## 実際の動作
エラーメッセージが表示されない
```

イシュー番号は`#2`とします。

---

#### ステップ2: ブランチを作成

```bash
git switch main
git pull origin main
git switch -c bugfix/2-fix-login-error
```

---

#### ステップ3: バグ修正

```bash
# コードを修正
git add .
git commit -m "Fix login error message #2"
git push origin bugfix/2-fix-login-error
```

---

#### ステップ4: プルリクエストを作成

GitHubでプルリクエストを作成します。

**タイトル**: ログインエラーが表示されない #2

**説明**:

```
## 概要
ログインエラーメッセージが表示されるように修正しました

## 変更内容
- エラーメッセージの表示処理を追加

Fixes #2
```

---

### 💡 TIP: イシューのラベル

イシューにラベルを付けると、**イシューの種類を分類**できます。

*   `bug`: バグ
*   `enhancement`: 新機能
*   `documentation`: ドキュメント
*   `question`: 質問

---

### 🚀 実践例3: イシューのラベルを使った分類

#### ステップ1: イシューを作成

GitHubで新しいイシューを作成します。

**タイトル**: READMEの更新

**説明**:

```
## 概要
READMEにインストール手順を追加する
```

---

#### ステップ2: ラベルを付ける

イシューに`documentation`ラベルを付けます。

---

### 💡 TIP: イシューのマイルストーン

イシューにマイルストーンを設定すると、**リリースごとにイシューをグループ化**できます。

*   `v1.0.0`: 最初のリリース
*   `v1.1.0`: 次のリリース

---

### 🚨 よくある間違い

#### 間違い1: イシューを作成せずに開発

```bash
# NG: イシューを作成せずに開発
git switch -c feature/add-login
```

**対処法**: 必ずイシューを作成してから開発します。

---

#### 間違い2: ブランチ名にイシュー番号を含めない

```bash
# NG: ブランチ名にイシュー番号を含めない
git switch -c feature/add-login
```

**対処法**: ブランチ名にイシュー番号を含めます。

```bash
# OK
git switch -c feature/1-add-login
```

---

#### 間違い3: コミットメッセージにイシュー番号を含めない

```bash
# NG: コミットメッセージにイシュー番号を含めない
git commit -m "Add login form"
```

**対処法**: コミットメッセージにイシュー番号を含めます。

```bash
# OK
git commit -m "Add login form #1"
```
