*   Excelの、「シート」「列」「行」との、対応関係を、理解する。

## 導入

前のセクションで、リレーショナルデータベースは、データを、「テーブル」という、表形式で、管理することを学びました。この「テーブル」は、データベースの、世界における、最も、基本的な、構成単位です。

このセクションでは、テーブルを、さらに、分解し、その、構成要素である、**「カラム」**と、**「レコード」**について、詳しく、見ていきます。これらの、用語は、SQLを、書いたり、データベースについて、誰かと、話をしたりする上で、避けては通れない、基本中の、基本です。Excelとの、対応関係を、意識しながら、正確に、理解していきましょう。

---

## 詳細解説

### 📊 テーブル、カラム、レコードの三位一体

顧客情報を、管理する、`customers`テーブルを、例に、見てみましょう。

**`customers` テーブル**

| `id` (顧客ID) | `name` (氏名) | `email` (メールアドレス) | `registered_at` (登録日) |
| :--- | :--- | :--- | :--- |
| 1 | 山田 太郎 | taro@example.com | 2023-01-15 |
| 2 | 鈴木 花子 | hanako@example.com | 2023-02-20 |
| 3 | 佐藤 次郎 | jiro@example.com | 2023-03-10 |

この表全体が、一つの、**テーブル（Table）**です。テーブルは、特定の、テーマに関する、データの、集まりです。（例: 顧客テーブル、商品テーブル、注文テーブル）

#### カラム（Column）

表の、**縦の列**、つまり、「`id`」「`name`」「`email`」「`registered_at`」といった、各項目が、**カラム（Column）**です。カラムは、そのテーブルが、どのような、データを持っているかを、定義します。カラムには、必ず、名前（カラム名）が、付けられます。

*   **カラム名**: `id`, `name`, `email`, `registered_at`

#### レコード（Record）

表の、**横の行**、つまり、一人一人の、顧客データが、**レコード（Record）**です。レコードは、テーブルに、格納されている、具体的な、データの、一件一件を、指します。例えば、`id`が`1`の、「山田 太郎」さんの、データが、一つの、レコードです。

*   **レコード1**: `(1, "山田 太郎", "taro@example.com", "2023-01-15")`
*   **レコード2**: `(2, "鈴木 花子", "hanako@example.com", "2023-02-20")`

### 🆚 Excelとの対応関係

これらの、用語は、Excelの、用語と、非常によく似ています。対応関係を、整理しておきましょう。

| データベース用語 | Excel用語 | 説明 |
| :--- | :--- | :--- |
| **テーブル (Table)** | **シート (Sheet)** | データの、集まり全体。 |
| **カラム (Column)** | **列 (Column)** | データの、項目・属性。 (例: 氏名、メールアドレス) |
| **レコード (Record)** | **行 (Row)** | 一件分の、具体的なデータ。 (例: 山田さんのデータ) |

> **補足**: データベースの世界では、カラムのことを、**フィールド（Field）**、レコードのことを、**タプル（Tuple）**と呼ぶこともありますが、意味は、同じです。本書では、主に、「カラム」と、「レコード」という、用語で、統一します。

### 🔢 データ型（Data Type）の重要性

データベースの、テーブルを、作成する際には、各カラムに、**「どのような、種類の、データを、格納するか」**を、あらかじめ、定義しておく必要があります。これを、**データ型（Data Type）**と呼びます。

Excelでは、一つの、列に、文字列も、数値も、日付も、自由に、入力できてしまいますが、データベースでは、これを、厳密に、区別します。

なぜなら、データ型を、正しく、設定することには、多くの、メリットがあるからです。

1.  **データの、整合性の、保証**: 例えば、「年齢」カラムに、`INTEGER`（整数）型を、設定しておけば、「二十歳」のような、文字列が、誤って、入力されるのを、防ぐことができます。システム的に、データの、品質を、保つことができるのです。
2.  **記憶領域の、効率化**: 例えば、「性別」のように、「男性」「女性」「その他」の、数種類しか、値が、入らない、カラムには、`VARCHAR`（可変長文字列）よりも、`ENUM`や、`TINYINT`（小さな整数）といった、より、メモリ消費の少ない、データ型を、使うことで、ディスクスペースを、節約できます。
3.  **計算や、比較の、正確性**: 「登録日」カラムが、`DATETIME`（日付時刻）型で、あれば、「2023年1月以降に、登録した、ユーザー」といった、日付に基づいた、検索や、計算が、正確かつ、高速に、行えます。もし、これが、単なる、文字列だと、このような、処理は、非常に、困難になります。

代表的な、データ型には、以下のようなものがあります。

| データ型 | 説明 | 例 |
| :--- | :--- | :--- |
| **`INTEGER`** (または `INT`) | 整数 | `1`, `100`, `-50` |
| **`VARCHAR(n)`** | 可変長の文字列（最大n文字） | `VARCHAR(255)` は、最大255文字の文字列を格納 |
| **`TEXT`** | 長い文字列 | 記事の本文など |
| **`DATETIME`** | 日付と時刻 | `2023-10-26 10:30:00` |
| **`DATE`** | 日付のみ | `2023-10-26` |
| **`BOOLEAN`** (または `BOOL`) | 真偽値（True / False） | `true`, `false` (内部的には 1 / 0) |

テーブルを、設計する際には、「このカラムには、どんなデータが、入るだろうか？」と、考え、最適な、データ型を、選択することが、非常に、重要です。

## ✨ まとめ

このセクションでは、データベースを、構成する、3つの、基本要素について学びました。

*   **テーブル**: データの、集まり全体。（Excelの**シート**に相当）
*   **カラム**: データの、項目・属性。（Excelの**列**に相当）
*   **レコード**: 一件分の、具体的なデータ。（Excelの**行**に相当）

そして、カラムを、定義する際には、**データ型**を、指定することが、重要であることも学びました。データ型は、データの、整合性を、保ち、記憶領域を、効率化し、正確な、計算を、可能にするための、重要な、仕組みです。

これらの、用語は、今後の、学習で、何度も、登場します。それぞれの、意味と、役割を、しっかりと、区別して、覚えておきましょう。
